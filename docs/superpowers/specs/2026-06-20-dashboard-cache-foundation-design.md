# Dashboard Cache Foundation Design

## Problem

SuperNova performance-summary endpoints currently calculate every response from PostgreSQL. Bulk Attendance and Orders KPI changes can make summaries expensive to rebuild, while individual Deduction and Annual Leave actions only affect one target user and must not trigger a global rebuild.

## Scope

Cache the canonical, unfiltered monthly performance summaries for active Pickers, Champs, and Area Managers. Admin workspace caching, workflow-rule changes, custom date-range caching, scoped-filter caching, and microservices are outside this change.

## Architecture

The API process emits domain events through `@nestjs/event-emitter` after the database mutation has committed. A cache event listener translates those events into jobs on a dedicated BullMQ `dashboard-cache` queue. The separate NestJS worker process consumes the jobs and calls `DashboardCacheWarmerService`.

Two job types are supported:

- `BULK`: generated by successful Attendance or Orders KPI confirmations after imported records become active. It cursor-paginates eligible users in fixed-size chunks and warms each requested month.
- `TARGETED`: generated by a Deduction final approval, Annual Leave final approval, or Annual Leave cancellation. It contains the affected `userId` and relevant month, resolves only that user, and never enumerates other users.

The event-to-queue boundary remains asynchronous so completed business transactions are not rolled back when Redis is unavailable. Listener failures are logged. Cache misses and the 15-minute TTL guarantee recovery through PostgreSQL calculation even if an in-process event is lost.

## Event Contract

Event names are centralized constants:

- `import.attendance.success`
- `import.kpi.success`
- `user.metrics.updated`

Every payload carries a stable domain `eventId` derived from the committed import batch or request. Bulk import payloads also carry one or more affected `YYYY-MM` months and the import source. Targeted payloads also carry `userId`, one affected `YYYY-MM` month, and the workflow source. Events are emitted only after successful final database state changes, never for preview or intermediate approval states.

## Queue Contract

The dedicated queue is named `dashboard-cache`. Jobs use discriminated payloads:

- Bulk job: `{ kind: "BULK", eventId: string, months: string[], source: "ATTENDANCE_IMPORT" | "KPI_IMPORT" }`
- Targeted job: `{ kind: "TARGETED", eventId: string, userId: string, month: string, source: "DEDUCTION" | "ANNUAL_LEAVE" }`

Job IDs are deterministic enough to coalesce duplicate updates for the same source and target window without merging unrelated users. Queue retries use bounded attempts and backoff. A targeted processor branch calls one targeted warmer method; only the bulk branch contains user pagination.

## Cache Contract

Keys use:

```text
perf_summary:{role}:{userId}:{month}
```

Roles are normalized to lowercase key segments. Values are JSON performance-summary responses with a 900-second TTL. Redis writes use `ioredis` pipelines: one pipeline per bulk chunk and one single-command pipeline for targeted warming.

Bulk warming selects active users with active accounts and employment status for roles that expose month-based performance summaries: `PICKER`, `CHAMP`, and `AREA_MANAGER`. Cursor pagination avoids loading all users into memory.

The key is used only for a canonical, unfiltered monthly query:

- `dateFrom` is the first calendar day of the key month.
- For a historical month, `dateTo` is the final calendar day of that month.
- For the current month, `dateTo` is the current UTC calendar date.
- Champ requests have no `vendorId`; Area Manager requests have no `chainId`.
- Picker requests use the matching monthly range, regardless of whether the optional period label is omitted or `THIS_MONTH`.

Last-week, quarter, custom, future, cross-month, branch-filtered, and chain-filtered requests bypass Redis and calculate directly from Prisma. This rule prevents different query variants from colliding under a month-only key.

## Calculation Boundaries

Workspace services expose cache-bypassing calculation methods and cache-aware public read methods:

1. Validate whether the request is the canonical unfiltered monthly query.
2. For a non-canonical request, calculate directly from Prisma without reading or writing Redis.
3. For a canonical request, attempt Redis read.
4. Return a valid cached payload on hit.
5. On miss, invalid JSON, or Redis failure, calculate from Prisma.
6. Attempt a cache write without failing the HTTP response if Redis is unavailable.

The worker invokes only cache-bypassing calculation methods, then writes the computed payloads. This prevents recursive reads of stale cache entries.

## Invalidation and Consistency

For targeted events, the listener invalidates the exact role/user/month key before enqueueing the warming job. If enqueueing fails, subsequent HTTP reads calculate from Prisma and repopulate the key. Bulk jobs overwrite affected month keys in chunks; they do not perform broad Redis key scans or global deletes that would create a cache stampede.

The design is eventually consistent because EventEmitter2 is process-local. A transactional outbox is the future upgrade if guaranteed post-commit event delivery becomes mandatory.

## Error Handling

- Redis read/write failures are logged and never convert a valid workspace request into a 500 response.
- Malformed cached JSON is deleted and treated as a miss.
- Missing or ineligible targeted users complete as a no-op with a structured worker result.
- One failed user calculation does not discard successful writes from the rest of a bulk chunk; failures are logged and reported in the job result.
- Worker jobs retry transient failures with bounded backoff.

## Testing

Focused tests will prove:

- Bulk import events enqueue only bulk jobs with affected months.
- Deduction and Annual Leave finalization events carry the exact target user and month.
- Targeted processing calculates and writes exactly one user without calling user enumeration.
- Bulk processing paginates users and writes with chunk pipelines.
- Workspace reads return cached values on hit.
- Custom, cross-month, and scoped-filter reads bypass Redis.
- Cache miss and Redis failure fall back to Prisma and attempt cache population.
- Invalid cached JSON is discarded and recalculated.
- Worker routing keeps bulk and targeted code paths separate.

The existing import, workflow, typecheck, lint, Prisma, and build checks remain required.
